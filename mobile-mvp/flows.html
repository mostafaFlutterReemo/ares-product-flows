PLAYER-JOURNEY PREREQUISITES

Admin setup referenced here

Before players start, an admin creates the cohort and cohort code, publishes the applicable habit/challenge, XP, bonus, badge, and rank versions, publishes EN/NL content, and sets the Daily Limit policy. Detailed dashboard flows and permissions are specified separately.

Admin steps required before the first player can start

  1. 01Create and activate the cohort with its program dates, timezone, capacity, and EN/NL availability.
  2. 02Set one active 6–12 character cohort code. The code uses unambiguous letters/numbers and may have an expiry or enrollment cap.
  3. 03Review or edit the launch-default catalog, exact habit/challenge XP, daily-target and full-set bonuses, rank thresholds, and badge definitions; publish the approved future-effective versions, assignments, and challenge occurrences.
  4. 04Set the cohort Daily Limit threshold and effective date.
  5. 05Publish the required onboarding, motivational, badge, and notification copy in English and Dutch.
Scope correction: “Fully GDPR-compliant” and “fully EU-hosted” are release outcomes that require DPO/legal and architecture review, not properties the UI or technology stack can claim automatically.

Canonical time rules

  • Program dates, not device calendar dates, drive reward slots and streaks. The server assigns the effective date.
  • The week is Monday–Sunday by program date. Day, week, and month leaderboards use that same date.
  • DST is resolved through the cohort’s IANA timezone. A timezone change is scheduled, migrated, and audited.
  • Day N equals 1 plus the number of program dates since the cohort start date; it is not the number of app opens.
REWARD INTEGRITY

Action to authoritative XP

The client requests an action. Only the backend validates and writes XP, levels, streaks, badges, and leaderboard projections.

User intent

Concurrency and failure behavior

  • Double taps and network retries reuse the same key and return the original result.
  • Concurrent devices compete for one unique completion record; only one award slot can be active.
  • Habit write failure rolls back optimistic state. Check-in failure preserves the draft.
  • The device clock and client-provided XP amount are ignored.
  • Corrections append reversal/adjustment events; history is not silently overwritten.
  • Revoked access closes the session and clears cached participant/cohort data.

POLICY V1

Admin-set, cohort-wide

  • Duration unit is minutes; allowed range 30–300 in 15-minute increments.
  • The editor prefills 60 minutes, but an authorized admin must explicitly publish the effective-dated cohort policy.
  • No per-user override. The active value is locked at program-day open and changes take effect at a future 02:00 boundary.
  • ARES youth UI displays only pending/final status. The native OS limit experience provides any real-time limit alert.
FINALIZATION V1

Conclusive or unavailable

  • One verified primary device is bound to a participant for each program day; changes start at the next 02:00 boundary.
  • A conclusive callback requires valid permission/entitlement and continuous native coverage for the policy window.
  • Identical retries return the prior result. Conflicting callbacks, a coverage gap, or missing callback by 04:00 become unavailable.
  • A signed late native callback may correct the same program date for seven days; it appends award/reversal events and never moves XP to the callback date.
Permission disclosure: monitoring is optional. Before the OS prompt, explain that enabling it can earn non-leaderboard lifetime XP and private badges; declining keeps every core feature available. An unavailable day freezes the within-limit streak—it does not increment or reset it. Admins cannot manually choose a result or mint the reward.
RIGHTS & ERASURE

Controlled deletion journey

Deletion is a resumable compliance workflow, not the same action as ordinary account deactivation.

Deletion effects

The account immediately stops login, assignments, notification targeting, and leaderboard presence. Only an approved request enters the idempotent delete/anonymize cascade and aggregate rebuild. A lawful hold remains locked and deletion-pending; a verified rejection or participant cancellation may restore the prior lifecycle state and is audited. Approved execution removes or irreversibly anonymizes Auth, profile, consent links, memberships, check-ins, completions, XP, badges, notifications, device registrations, Daily Limit results, and media. Partial failure remains retryable and cannot reactivate the account. Normal completion target: within 30 days, subject to a documented lawful exception.

Reward settlement rules

DAILY ACTIONS

The server posts each habit's day-snapshotted XP and the snapshotted target/full-set bonuses. Launch defaults are +5 per habit, +50 at six eligible habits, and another +50 for the complete required set. Morning and evening check-ins remain +5 each, with no pair bonus.

MISSIONS

Each occurrence shows and awards one exact snapshotted value from its published challenge version. Launch defaults are +30 for a daily challenge and +400 for a weekly mission.

DAILY LIMIT

A finalized Within limit result posts one atomic +130 XP event under the MVP v1 ruleset. Exceeded or Unavailable posts zero.

Server authority: celebrations never determine eligibility. The backend writes a unique ledger event first; any Claim or View action is idempotent and purely presentational. Badges and level-ups award 0 XP.

Notification deep links

Push notification taps route to specific screens. Not just the inbox.

Deep-link and delivery policy

  • Check-in reminders open Check-in; habit reminders open Habits; challenge reminders open the relevant challenge occurrence.
  • Streak, badge, and level events open Progress; position changes open Leaderboard.
  • The daily-limit push says only “Your daily limit status is ready.” The lock screen never reveals the result.
  • Admin announcements open a read-only Inbox detail. Mentor chat is not part of MVP.
  • Every event creates an Inbox record even when push permission is denied.
  • Deep links are chosen from a fixed allow-list; arbitrary URLs are rejected.
  • Quiet hours are 21:00–07:00 in the cohort timezone, with a maximum of three non-transactional pushes per participant/day.
  • Expired, removed, or unauthorized destinations return to Inbox with neutral copy.
Inbox behavior: rename “Clear All” to “Mark All Read.” Inbox records are kept indefinitely unless the account is deleted, and achievement messages caused by one action should be bundled.

ARES STREAK

Configured-target consistency chain

  • A day qualifies only after finalization when the snapshotted target count of distinct streak-eligible assigned habits is complete. The launch count is six; H10 is launch-default ineligible.
  • Today remains provisional. Store current streak, longest streak, last qualified date, and shield balance.
  • Earn one shield at 7, 30, 60, 90, then every additional 30 real qualified days; maximum balance is one.
  • A missed day consumes a shield and preserves the chain but grants no day credit or XP; without a shield, current streak resets.
OTHER STREAKS

Independent, transparent measures

  • Individual habit streaks use their own due schedule and no shield.
  • The within-limit streak counts only finalized within_limit days, offers no XP bonus, and uses no shield.
  • unavailable never counts as within limit, never creates an exceeded label, and freezes rather than resets the within-limit streak.
  • Repair is reserved for verified outage, synchronization, or admin error and requires actor, reason, evidence, before/after values, and timestamp.
Ledger contract: every event stores user, organization, cohort-at-event, source definition/version/occurrence and snapshot IDs, program date/week, server timestamp, the exact awarded lifetime and leaderboard deltas, ruleset version, idempotency key, and optional reversal reference. Corrections append an exact compensating event against the original snapshot. Badges, ranks, and Daily Limit results never create hidden bonus XP.

Authorized admins can create and version supported badge names, EN/NL copy, icons, tiers, and structured triggers. The table above is the launch-default version. A published definition is immutable; criteria changes apply prospectively and every historic award retains its badge-version and evidence snapshot. Every badge is idempotent, awards 0 XP, creates an Inbox message, and is revoked only when an erroneous source event is corrected. Badge evidence and its source records are kept indefinitely unless the account is deleted; Daily Limit-derived details remain excluded from external profile sharing.

Cohort leaderboard calculation

  • Day, Week, and Month sum only events whose immutable source snapshot stores leaderboardEligible=true; lifetime rank appears only as context.
  • Competition ranking uses ties such as 1, 2, 2, 4 with an opaque stable tie key for display order.
  • Only opted-in safe aliases appear; opting out never changes earned XP.
  • Hide the board until at least five opted-in cohort members are visible.
  • Never show a global board, surname, bottom-participant list, or “XP behind” pressure message.
  • Deactivated and deleted accounts disappear; historic source events stay attributed for authorized aggregates.

Cohort-code transaction: store a salted hash of the normalized code and show its raw value only when the admin creates or rotates it. Daily Limit data contract

DAILY LIMIT POLICY

Authorized configuration

  • organizationId, cohortId, thresholdMinutes (30–300, step 15), and IANA timezone.
  • effectiveFrom/effectiveTo, policy version, feature enabled, created/published actor and timestamps. Reward value is not editable here.
  • The policy active at day-open is locked for that program date; a change applies from the next scheduled boundary.
  • The configured threshold is policy data visible only to authorized administrators editing the cohort.
  • The MVP v1 +130 value and its leaderboard exclusion live in the approved gamification ruleset, separate from the Daily Limit policy editor.
DAILY LIMIT RESULT

Minimum final outcome

  • userId, organization, cohort-at-event, program date, policy version.
  • status: within_limit | exceeded | unavailable.
  • Server finalizedAt, bound primary-device ID, callback deadline, and coarse technical availability reason where operationally required.
  • Unique user/program-date constraint and source attestation metadata without usage detail; corrections reference the prior result/version.
Prohibited by design: measured or remaining duration, percentage, time saved, threshold overage, app/package/bundle names, installed app list, category, domain, content, browsing history, per-app events, and raw platform usage payloads must not appear in the schema, logs, analytics, crash reports, exports, admin UI, or participant UI.
Storage policy: ARES applies no automatic time-based deletion to player records. Accounts, check-ins, messages, completions, XP, badges, Daily Limit results, and audit evidence are kept indefinitely unless an explicit account-deletion workflow is completed.
