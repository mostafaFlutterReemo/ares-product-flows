AD-F02–03 · PROGRAM SETUP

Cohort creation, activation, and cohort-code enrollment

The cohort is the admin’s main operating boundary. Its timezone, assignments, Daily Limit policy, and code determine what a new player receives.

ACTOR · SUPER / PROGRAM ADMINTRIGGER · NEW PROGRAMPLAYER IMPACT · ELIGIBILITY + ASSIGNMENTS

Cohort setup requirements

  1. 01Create a draft with name, immutable internal ID, city/region if needed for reporting, IANA timezone, program start and end dates, capacity, and enrollment eligibility.
  2. 02Review or edit the launch-default habits, exact XP values, daily-target count/eligibility/bonus, required full set and bonus, daily challenge schedule, weekly mission catalog, rank thresholds, badges, check-in prompts, weekly motto, motivational copy, and announcement defaults.
  3. 03Set the Daily Limit threshold and future effective program date. The threshold is policy data; no observed usage time is received.
  4. 04Run readiness validation: complete EN/NL copy, at least six assigned non-H10 habits, valid schedule, approved safety review, capacity, date order, and no conflicting effective versions.
  5. 05Activate. Archive later to stop new enrollment and future assignments while keeping historic memberships, events, and reporting intact.

Cohort-code lifecycle

CREATE / ROTATE

One active reusable code

  • Use 6–12 unambiguous letters and numbers; trim and normalize to uppercase.
  • Optional expiry and enrollment cap; status is active, disabled, expired, capacity reached, or rotated.
  • Show raw code only in create/rotate success; store a salted hash and never include the raw value in logs or exports.
  • Rotation immediately blocks the prior code for new reservations.
VALIDATE / CONSUME

Atomic and enumeration-safe

  • Check cohort lifecycle, effective dates, capacity, enrollment/age policy, and attempt limits.
  • A valid check reserves capacity for 30 minutes; registration revalidates and consumes once.
  • Five failed attempts in 15 minutes trigger progressive cooldown with a neutral error.
  • Existing players cannot self-transfer with a second code.
No invitation transaction exists: the admin can copy a code for the organization’s existing offline distribution process, but ARES does not email, text, or push an invitation and does not create pending invited users.

REQUIRED FIELDS

  • Shaming, judgmental, surveillance-like, or “no excuses” language.

AD-F08 · ANNOUNCEMENTS

Compose, validate, schedule, deliver, and review

The manual composer sends one-way admin announcements. Automated reminders and achievement notifications are server-owned templates and cannot be spoofed by an admin.

ACTOR · SUPER / PROGRAM ADMINAUDIENCE · ALL / COHORT / ONE USERSOURCE OF TRUTH · INBOX

Main path

  1. 01Select All organization participants, one cohort, or one active participant. Show the server-calculated eligible recipient count and exclusions.
  2. 02Enter localized title and body: maximum 60 characters for title and 240 for body. All/cohort audiences require complete EN and NL variants.
  3. 03Select a destination from an allow-list: Inbox detail, Home, Habits, Daily Check-in, Challenges, Progress, Leaderboard, Profile, or Daily Limit. Arbitrary URLs and typed route paths are rejected.
  4. 04Preview each locale, lock-screen-safe fallback copy, audience, schedule, and recipient count. All-user sends require typed confirmation.
  5. 05Create an Inbox record for every eligible recipient. Attempt push only when permission/token/quiet-hours rules allow; push delivery never determines whether the announcement exists.
  6. 06Show campaign history, totals, failures, and retry eligibility without exposing another participant’s private delivery data.
MANUAL TYPE

Admin announcement only

  • One-way, read-only Inbox detail; players cannot reply.
  • Title/body cannot claim a badge, level, rank, streak, XP, or Daily Limit result.
  • Quiet hours: 21:00–07:00 in the cohort timezone.
  • Maximum three non-transactional pushes per participant per program day.
AUTOMATED TYPES

Template and history view

  • Morning/evening reminders, habit/challenge reminders, Daily Limit result-ready, streak warning, level-up, badge, and weekly summary are system events.
  • Admins may manage approved localized templates, but cannot manually trigger a false achievement.
  • Every system event also creates an Inbox record.
  • Transactional safety/rule messages bypass marketing-rate caps but still use safe copy.

ANALYTICS

  • Keep identity, check-in-answer, and aggregate export permissions separate.
  • Show loading, no data, filtered empty, suppressed, stale, partial, permission denied, and error states.

CSV export contract

  1. 01Admin chooses an allow-listed report, scope, period, and approved fields. Individual check-in-answer export requires explicit permission and a recorded purpose.
  2. 02Server snapshots filters, metric versions, role, and membership-at-event, then queues an asynchronous job.
  3. 03Apply suppression, remove non-authorized columns, escape CSV formula prefixes, encode UTF-8, and encrypt the output.
  4. 04Notify the requesting admin when ready. Download requires a current authorized session and permission for the requested individual data.
  5. 05The temporary download expires after 24 hours. Source records and export audit metadata remain stored indefinitely under the normal storage policy.
Neutral language: remove “streak losers,” “at risk,” “mentors need to reach out,” peer-passing activity, and bottom rankings. Report observable events—such as “no confirmed activity in 14 days”—without predicting character, motivation, safety, or dropout.
AD-F10–12 · PRIVACY, POLICY & AUDIT

Every sensitive action is authorized, minimized, and explainable

The dashboard operates youth data under least privilege. Normal records are stored indefinitely; a verified account-deletion workflow is the explicit exception.

Daily Limit policy administration

POLICY DATA

Effective-dated cohort rule

  • Cohort, threshold minutes, IANA timezone, effective program date, status, creator, publisher, and version.
  • One effective version at a time; replacement starts at a future 02:00 boundary.
  • Show affected cohort, policy version, and linked gamification version before publishing.
  • The Daily Limit-linked habit uses its exact HabitVersion XP; the separate +130 reward remains the launch default unless its ruleset field is explicitly changed.
RESULT DATA

Three values, no raw usage

  • within_limit, exceeded, or unavailable, plus program date, version, and finalization evidence.
  • No observed minutes, remaining minutes, progress percentage, installed apps, app names, categories, domains, content, or browsing history.
  • Admin correction is reserved for verified processing error and cannot invent a within-limit status.
  • Unavailable freezes the within-limit streak and awards nothing.

Trust the current server state

  • Loading skeleton, populated, empty, filtered empty, stale/last updated, permission denied, service error, and retry.

  • URL filters are validated and cannot broaden scope.
EVERY EDIT

Safe concurrency

  • Dirty-state warning, field validation, saving, saved, server rejection, version conflict, and retry.
  • Optimistic concurrency uses the record version; stale writes never overwrite silently.
  • Published/referenced data creates a new effective version.
EVERY SENSITIVE ACTION

Explicit consequence

  • Permission, purpose, player effect, typed confirmation or distinct approval where required, idempotent result, and audit.
  • Disabled control includes a human-readable reason.
  • Partial jobs remain visible and retryable.
